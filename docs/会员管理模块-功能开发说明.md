# 会员管理模块 — 功能开发说明（验收版）

**读者**：产品、后端、前端、测试。  
**性质**：各菜单的**功能范围、业务规则与验收标准**；开发与测试按本文做用例与对照，接口字段名与错误码以接口文档为准。

---

## 一、模块总览

本模块覆盖后台 **会员管理** 菜单下共 **9** 个功能入口，用于配置 VIP、信用分、身份认证与出金策略。

| 序号 | 功能名称 | 访问路径 |
|------|----------|----------|
| 1 | VIP 等级配置 | `/admin/users/vip-config` |
| 2 | VIP 升级日志 | `/admin/users/vip-upgrades` |
| 3 | 信用分配置 | `/admin/users/credit-score-config` |
| 4 | 信用分变动审核 | `/admin/users/credit-score-audit` |
| 5 | 信用分变动日志 | `/admin/users/credit-score-changes` |
| 6 | 认证等级配置 | `/admin/users/verification-config` |
| 7 | 认证身份审核 | `/admin/users/verification-audit` |
| 8 | 认证审核日志 | `/admin/users/verification-log` |
| 9 | 出金策略 | `/admin/users/withdraw-policy` |

**跨模块统一要求**

| 项目 | 验收标准 |
|------|----------|
| 权限 | 仅授权管理员可访问；审核、保存策略等操作须记录操作人及时间（后端落库）。 |
| 枚举一致 | VIP 等级、认证三档、信用分区间与出金策略、信用分配置中的命名一致，否则规则无法命中。 |
| 列表分页 | 日志与审核列表须支持服务端分页（页码、每页条数、总条数）；筛选条件须参与查询。 |
| 审核操作 | 同一单据不得重复生效；须幂等；非法状态不得提交。 |
| 时间 | 展示时间与时区规则在全站统一；筛选「日期范围」须对边界定义一致（含起止日的 0 点～24 点或等价 UTC）。 |

## 二、分模块说明

### 1. VIP 等级配置

**目标**：维护 VIP 阶梯及展示信息，供前台与升级逻辑使用。

**数据项（须全部可配置或展示）**

| 项 | 规则 |
|----|------|
| 等级编号 | 非负整数；编辑时**不可修改**。 |
| 系统名称 | 必填或按业务必填规则。 |
| 显示名称 | **必填**。 |
| 前台徽章图标 | 仅允许图片类型；单文件 **≤ 1MB**；上传后须预览；存储地址由服务端返回。 |
| 最低信用分 | 整数，范围 **0～800**（与当前表单校验一致）。 |
| 状态 | **启用 / 禁用** 二选一。 |
| 等级说明 | 选填，长文本。 |

**业务规则**

- **等级 0（普通用户）不得删除、不得切换为禁用**；其余等级可删除（须二次确认）、可启用/禁用。
- 列表须按等级编号 **升序** 排列。
- **会员权益明细**不在本页维护；本页不编辑权益列表字段。

**验收**

- 违反上述删除/禁用限制时须拦截并提示。
- 展示名称空保存须失败并提示。
- 非图片或超 1MB 须拦截。

---

### 2. VIP 升级日志

**目标**：查询用户 VIP 等级变更历史。

**列表字段（须包含）**

用户标识、升级前等级、升级后等级、升级原因（信用分达标 / 充值升级 / 手动升级等）、关联业务信息（如充值金额，有则展示）、发生时间。

**筛选**

须支持：关键词（用户名、用户 ID、备注等约定字段）、升级原因、升级前等级、升级后等级、时间范围。

**分页**

须服务端分页；列表排序为 **升级时间倒序**。

**统计区**

- **口径须与产品定稿一致**：要么统计数字与**当前筛选结果**一致，要么明确为**全库全量**；两种不得混用且不标注。
- 各升级原因分类条数之和须等于「总升级记录」定义下的总条数（分类互斥时）。

**导出**

若本期上线导出：须支持按当前筛选条件导出，字段列表固定；若本期不上线，菜单或按钮不得误导为已可用。

**验收**

- 筛选与列表、总条数一致；翻页无重复、无遗漏。

---

### 3. 信用分配置

**目标**：配置信用分获取、扣减、上下限、VIP 联动与人工审核规则。

**配置分区**

须包含至少：**基础配置、获取规则、扣除规则、人工审核**（与界面 Tab 一致）。

**保存前校验（须满足）**

| 校验项 | 条件 |
|--------|------|
| 初始分数 | ∈ **[0，最高分]** |
| 充值换算基数（每多少 U 算 1 分） | **≥ 0** |
| 自动升级 VIP1 | 开关开启时，「升级到 VIP1 所需信用分」须 **> 0** |

**试算器（配置页内预览，须与下列公式一致）**

1. 得分从 **初始分** 开始。  
2. **充值加分** = **⌊ 试算充值金额 ÷ 配置的每 U 基数 ⌋**（向下取整）。  
3. **初级认证加分**、**高级认证加分**：勾选则加上对应配置分。  
4. **扣除规则开启时**：勾选违规 / 异常交易 / 纠纷等，分别减去对应扣分。  
5. **最终结果** = **min(最高分, max(最低分, 上述累加结果))**。

**人工审核（配置含义）**

须同时满足才进入人工审核队列：**变动类型 ∈ 已勾选的需审核类型**，且 **|变动分数| ≥ 审核阈值**。信用分系统总开关业务上视为**常开**，不在此页提供关闭。

**验收**

- 未保存离开须有提示；保存成功须有反馈。
- 试算结果与上式逐条核对；校验失败须有明确文案。

### 4. 信用分变动审核

**目标**：处理须人工审批的信用分变动。

**列表数据范围**

列表须只展示 **审核状态为「待审核」** 的申请（与筛选「状态」若并存时，以接口约定为准；**不得**在无说明的情况下展示已通过/已拒绝混入待办）。

**列表与详情字段**

须含：用户标识、变动类型、变动前后分数、变动量、申请时间、原因说明等；详情须可查看完整信息。

**操作**

- **通过**：变动按业务规则生效；须写入审核人、审核时间、审核备注（可选填）。  
- **拒绝**：变动**不生效**；须写入审核人、审核时间、拒绝说明（必填或按规则）。

**统计区**

待审核数、已通过数、已拒绝数、总申请数；通过率 = 已通过 ÷ 总申请 × 100%，拒绝率 = 已拒绝 ÷ 总申请 × 100%；总申请为 0 时比例显示 **0%** 或 **「—」**，全站统一。

**详情辅助展示**

变动幅度占变动前分数比例 = **|变动量| ÷ 变动前分数 × 100%**；变动前为 0 时**不得除零**，显示 **0%** 或 **「—」**，全站统一。

**验收**

- 同一单据重复审核须拒绝或提示；通过后用户信用分与日志一致。

---

### 5. 信用分变动日志

**目标**：查询已落库的信用分变动流水（**仅已生效记录**，不含仅申请未生效数据）。

**列表字段**

须含：用户、变动类型、变动前后分数、变动量、操作方（系统/管理员）、时间；有关联金额或单号时须展示。

**筛选**

关键词、变动类型、时间范围；**分页**；默认 **时间倒序**。

**统计区（固定口径）**

| 指标 | 定义 |
|------|------|
| 总变动记录 | 当前筛选条件下的条数 |
| 增加记录数 | 变动量 **> 0** 的条数 |
| 减少记录数 | 变动量 **< 0** 的条数 |
| 累计增加 | 所有 **变动量 > 0** 的变动量**求和** |

统计须与**同一筛选条件**下列表总条数及分类逻辑一致。

**验收**

- 与「审核」模块边界清晰：日志**不得**出现未审核通过即不应生效的变动。

---

### 6. 认证等级配置

**目标**：按 **未认证 / 初级认证 / 高级认证** 三档配置权限与证件，**不出金限额**。

**每档须可配（布尔）**

| 开关 | 说明 |
|------|------|
| 允许入金 | 该档是否允许入金 |
| 允许出金 | 该档是否允许出金 |
| 允许现货交易 | 该档是否允许现货 |
| 允许交割合约 | 该档是否允许交割合约 |
| 允许永续合约 | 该档是否允许永续合约 |

**证件**

- 高级认证：须可勾选所需证件类型列表。  
- 未认证 / 初级：展示证件要求说明；无多选时展示「无需」或列表为空。

**与出金策略的边界**

- **单笔最低出金、每日出金上限** 仅在 **出金策略** 配置；本页**不得**配置金额类限额。

**验收**

- 三档独立保存、独立展示；保存后回读一致。

---

### 7. 认证身份审核

**目标**：处理用户 **KYC 升级申请**。

**列表**

须含：用户标识、当前认证等级、申请等级、提交时间、状态；支持关键词、申请等级、时间范围筛选；**分页**；**申请时间倒序**。

**详情**

须展示基础资料与已上传证件列表；支持图片/PDF 等格式预览（按实现支持类型）。

**审核结论（须实现且互斥）**

- **通过**：用户认证等级更新为申请等级；记录审核人、时间。  
- **拒绝**：等级不变；须记录原因。  
- **要求补件**：进入补件状态；用户可再次提交（若业务流程包含补件）。

**验收**

- 状态机合法；无越权从终态再次提交。

---

### 8. 认证审核日志

**目标**：查询认证审核相关操作，满足审计。

**记录范围**

须包含审核通过、审核拒绝、要求补件等类型；**配置变更类日志**是否纳入本页须单独约定，未纳入则不得混入。

**筛选**

关键词（用户名、用户 ID、操作人、描述）、操作类型、日期范围；**分页**；**操作时间倒序**。

**统计**

| 指标 | 定义 |
|------|------|
| 今日日志 | 操作时间落在 **当前自然日**（业务时区）内的条数 |
| 总日志数 | 本页展示类型下的全部条数（或当前筛选下，须标注口径） |
| 审核操作次数 | 通过+拒绝+补件等审核类动作次数（定义固定） |

**验收**

- 日期筛选含边界；关键词大小写规则固定（如统一转小写匹配）。

### 9. 出金策略

**目标**：配置 **U 本位** 单笔最低出金与每日出金上限，并与 VIP、认证、信用分三维度联动。

**全局默认**

当下列任一情况成立时，使用全局默认的 **单笔最低** 与 **每日上限**：  
① 所有维度均关闭；② 已开启维度但**全部未匹配到规则**。

**单维度匹配规则**

| 维度 | 匹配方式 |
|------|----------|
| VIP | 用户当前 VIP 等级与表格中 **等级编号** 完全一致，取该行。 |
| 认证 | 用户当前认证档位与表格中 **档位** 完全一致，取该行。 |
| 信用分 | 用户信用分 **∈ [区间下限，区间上限]**（闭区间），命中唯一一行；区间须**不重叠**，重叠时须报错或拒绝保存（由实现选定一种并写死）。 |

**多维度合并（最严格模式，固定算法）**

对每个**已开启**且**已匹配到行**的维度，得到该维度的 `单笔最低`、`每日上限`（「无上限」记为 null）。

- **生效单笔最低** = 各维度 `单笔最低` 的 **最大值**。  
- **生效每日上限**：仅对 **非 null** 的每日上限取 **最小值**；若**全部**为 null，则生效为 **无上限**。

维度在界面上的**排序仅影响展示**，**不得**参与合并计算（不得采用「按优先级取第一条」覆盖最严格合并）。

**试算**

输入 VIP、认证等级、信用分，须即时算出 **生效单笔最低、生效每日上限**，并与保存后的策略一致（未保存试算对象须明确为草稿或仅前端预览）。

**与 VIP 配置联动**

VIP 等级增删后，出金策略中 **按 VIP 维度的行** 须与当前 VIP 列表同步（多余行删除，新等级补默认行），已编辑等级的数值在同步规则下保留或按约定重置。

**验收**

- 手工构造多组 VIP+认证+信用分，核对合并结果与上式一致。  
- 「无上限」与数值混合时，每日上限须为数值中的最小值。

## 三、测试用例索引（建议）
